# 秒杀系统数据库消费与批量写入优化方案

> MQ 消费者批量落库：建订单 + 扣真实库存，唯一键防重

## 一、现有问题分析

### 1.1 当前订单表缺少唯一键

```sql
-- 当前 flash_order 表结构
CREATE TABLE `flash_order` (
    `id`           BIGINT AUTO_INCREMENT PRIMARY KEY,
    `order_id`     VARCHAR(64)  NOT NULL,
    `fs_id`        VARCHAR(32)  NOT NULL,
    `uid`          VARCHAR(64)  NOT NULL,
    `goods_info_id` VARCHAR(64),
    `flash_price`  DECIMAL(10,2),
    `qty`          INT DEFAULT 1,
    `order_status` INT DEFAULT 0,    -- 0=待支付 1=已支付 2=已取消
    `create_time`  BIGINT,
    `pay_time`     BIGINT,
    -- 缺少: request_id 字段
    -- 缺少: 唯一约束 (uid, fs_id)
    INDEX idx_uid_fs(uid, fs_id),
    INDEX idx_status_time(order_status, create_time)
);
```

| 问题 | 风险 |
|---|---|
| **无唯一键防重** | 重复 MQ 消息会导致重复订单 |
| **逐条 insert** | 每次 insert 都要等 DB 确认，吞吐上不去 |
| **扣库存与建订单非事务** | 订单建了但库存没扣，或相反 |
| **无 request_id** | 无法幂等 |

### 1.2 当前消费流程

```
MQ消息 → 逐条 insert → 单条 insert
          每条1次RTT       吞吐瓶颈 ~1200 TPS
```

### 3.3 设计要点

| 设计 | 说明 |
|---|---|
| **`uk_uid_fs` 唯一键** | 同一用户对同一活动只能有一条记录，DB 层面保证不重复 |
| **`uk_request_id` 唯一键** | MQ 消息级别幂等，重复消息 insert 时直接报 DuplicateKey 错误 |
| **`version` 乐观锁** | 扣减真实库存时 `UPDATE ... SET sold_stock=sold_stock+1, version=version+1 WHERE version=旧值` |
| **索引精简** | 写多读少，索引过多影响插入性能，只保留必要索引 |

## 四、批量消费核心实现

### 4.1 批量消费者

```java
/**
 * FlashBatchConsumer.java — 批量消费 MQ 写入 DB
 *
 * 核心设计：
 * 1. 攒批：每 100 条 或 每 500ms 刷一次
 * 2. 批量 insert + 批量 update 在同一个事务
 * 3. 唯一键冲突 = 重复消息，静默忽略
 */
@Component
@RocketMQMessageListener(
    topic = MqTopic.FLASH_ORDER,
    consumerGroup = "consumer-flash-order",
    consumeMode = ConsumeMode.ORDERLY,
    consumeThreadMax = 64,
    consumeThreadNumber = 32,
    batchSize = 32
)
public class FlashBatchConsumer implements RocketMQListener<MessageExt> {

    private static final Logger log = LoggerFactory.getLogger(FlashBatchConsumer.class);

    @Autowired
    private BatchOrderService batchOrderService;

    // 每批次最大条数
    private static final int BATCH_SIZE = 100;
    // 最大等待时间
    private static final long FLUSH_INTERVAL_MS = 500;

    // 每个 fsId 一个缓冲区（顺序消费保证同一 fsId 进入同一线程）
    private final ConcurrentHashMap<String, BatchBuffer> buffers = new ConcurrentHashMap<>();

    // 定时刷新线程
    private final ScheduledExecutorService scheduler = Executors.newSingleThreadScheduledExecutor();

    @PostConstruct
    public void init() {
        scheduler.scheduleAtFixedRate(this::flushAll, FLUSH_INTERVAL_MS, FLUSH_INTERVAL_MS, TimeUnit.MILLISECONDS);
    }

    @Override
    public ConsumeOrderlyStatus onMessage(List<MessageExt> msgs) {
        for (MessageExt msg : msgs) {
            try {
                String body = new String(msg.getBody(), StandardCharsets.UTF_8);
                FlashOrderMessage message = JSON.parseObject(body, FlashOrderMessage.class);

                // 写入缓冲区
                BatchBuffer buffer = buffers.computeIfAbsent(
                    message.getFsId(), k -> new BatchBuffer(BATCH_SIZE, FLUSH_INTERVAL_MS));

                buffer.add(message);

                // 达到批量阈值，立即刷出
                if (buffer.size() >= BATCH_SIZE) {
                    flushBuffer(message.getFsId(), buffer);
                }

            } catch (Exception e) {
                log.error("消费解析失败: msgId={}", msg.getMsgId(), e);
                return ConsumeOrderlyStatus.SUSPEND_CURRENT_QUEUE_A_MOMENT;
            }
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }

    /** 定时刷出所有缓冲区 */
    private void flushAll() {
        buffers.forEach((fsId, buffer) -> {
            if (buffer.isExpired() && !buffer.isEmpty()) {
                flushBuffer(fsId, buffer);
            }
        });
    }

    /** 刷出单个缓冲区 */
    private void flushBuffer(String fsId, BatchBuffer buffer) {
        List<FlashOrderMessage> batch = buffer.drain();
        if (batch.isEmpty()) return;

        long start = System.currentTimeMillis();
        try {
            FlashOrderWriteResult result = batchOrderService.batchCreateOrders(batch);

            log.info("批量写入完成: fsId={}, batch={}, success={}, dup={}, fail={}, cost={}ms",
                fsId, batch.size(), result.getSuccessCount(),
                result.getDuplicateCount(), result.getFailCount(),
                System.currentTimeMillis() - start);

        } catch (Exception e) {
            log.error("批量写入异常: fsId={}, batch={}", fsId, batch.size(), e);
            // 批量失败 → 逐条重试
            batch.forEach(msg -> retrySingle(msg));
        }
    }

    /** 逐条重试（批量失败后的降级） */
    private void retrySingle(FlashOrderMessage msg) {
        try {
            batchOrderService.singleCreateOrder(msg);
        } catch (DuplicateKeyException e) {
            log.warn("重复订单, 忽略: requestId={}", msg.getRequestId());
        } catch (Exception e) {
            log.error("单条重试失败, 加入延迟队列: requestId={}", msg.getRequestId(), e);
            flashMqProducer.sendToDelayQueue(msg.getFsId(), msg.getUid(), msg.getRequestId(), 3);
        }
    }
}
```

### 4.2 缓冲区

```java
/**
 * BatchBuffer — 线程安全的批量缓冲区
 */
public class BatchBuffer {

    private final List<FlashOrderMessage> messages = new ArrayList<>();
    private final int maxSize;
    private final long maxWaitMs;
    private long lastFlushTime = System.currentTimeMillis();

    public BatchBuffer(int maxSize, long maxWaitMs) {
        this.maxSize = maxSize;
        this.maxWaitMs = maxWaitMs;
    }

    public synchronized void add(FlashOrderMessage msg) {
        messages.add(msg);
    }

    public synchronized List<FlashOrderMessage> drain() {
        List<FlashOrderMessage> drained = new ArrayList<>(messages);
        messages.clear();
        lastFlushTime = System.currentTimeMillis();
        return drained;
    }

    public synchronized int size() {
        return messages.size();
    }

    public synchronized boolean isEmpty() {
        return messages.isEmpty();
    }

    public boolean isExpired() {
        return (System.currentTimeMillis() - lastFlushTime) >= maxWaitMs;
    }
}
```

## 五、批量写入核心服务

### 5.1 批量创建订单 + 扣库存

```java
/**
 * BatchOrderService.java — 批量创建订单 + 扣真实库存
 *
 * 一个事务内完成：
 *   - 批量 INSERT flash_order（唯一键防重）
 *   - 批量 UPDATE flash_sale 库存（乐观锁防超卖）
 */
@Service
public class BatchOrderService {

    private static final Logger log = LoggerFactory.getLogger(BatchOrderService.class);

    @Autowired
    private FlashOrderMapper flashOrderMapper;

    @Autowired
    private FlashSaleMapper flashSaleMapper;

    /**
     * 批量创建订单（事务内）
     *
     * 先全部 INSERT，再批量 UPDATE 库存
     * 唯一键冲突 = 重复消息，不计入失败
     */
    @Transactional(rollbackFor = Exception.class)
    public FlashOrderWriteResult batchCreateOrders(List<FlashOrderMessage> messages) {
        FlashOrderWriteResult result = new FlashOrderWriteResult();

        // ---- 阶段1: 批量 INSERT 订单 ----
        // 拆分为"新订单"和"重复订单"
        List<FlashOrder> newOrders = new ArrayList<>(messages.size());
        Map<String, Integer> fsIdQtyMap = new HashMap<>();  // 统计每个活动需要扣减的库存

        for (FlashOrderMessage msg : messages) {
            try {
                FlashOrder order = buildOrder(msg);
                flashOrderMapper.insertWithUniqueCheck(order);  // INSERT IGNORE / ON DUPLICATE KEY
                newOrders.add(order);

                // 统计库存扣减数
                fsIdQtyMap.merge(msg.getFsId(), 1, Integer::sum);
                result.addSuccess();

            } catch (DuplicateKeyException e) {
                // UK_UID_FS 冲突 — 该用户已下过单，这是重复消息
                log.debug("重复订单, 忽略: uid={}, fsId={}, requestId={}",
                    msg.getUid(), msg.getFsId(), msg.getRequestId());
                result.addDuplicate();
            }
        }

        // ---- 阶段2: 批量扣减真实库存 ----
        if (!fsIdQtyMap.isEmpty()) {
            for (Map.Entry<String, Integer> entry : fsIdQtyMap.entrySet()) {
                int affected = flashSaleMapper.deductStock(entry.getKey(), entry.getValue());
                if (affected == 0) {
                    // 库存不足（理论上不会发生，Redis 预扣过了）
                    log.error("真实库存不足, 需要补偿: fsId={}, qty={}", entry.getKey(), entry.getValue());
                    // 触发告警，人工介入
                    alertService.alertStockShortage(entry.getKey());
                }
            }
        }

        return result;
    }

    /**
     * 单条创建订单（批量失败后降级）
     */
    @Transactional(rollbackFor = Exception.class)
    public void singleCreateOrder(FlashOrderMessage msg) {
        // 1. INSERT 订单（唯一键保护）
        flashOrderMapper.insertWithUniqueCheck(buildOrder(msg));

        // 2. 扣库存
        int affected = flashSaleMapper.deductStock(msg.getFsId(), 1);
        if (affected == 0) {
            log.error("单条扣库存失败: fsId={}", msg.getFsId());
            throw new BizException("库存不足");
        }
    }

    /**
     * FlashOrderMessage → FlashOrder
     */
    private FlashOrder buildOrder(FlashOrderMessage msg) {
        FlashOrder order = new FlashOrder();
        order.setRequestId(msg.getRequestId());
        order.setOrderId(generateOrderId(msg.getFsId(), msg.getUid()));
        order.setFsId(msg.getFsId());
        order.setUid(msg.getUid());
        order.setGoodsInfoId(msg.getFsId());      // 暂用 fsId，正式应从活动配置读取
        order.setFlashPrice(BigDecimal.valueOf(0.01));
        order.setQty(1);
        order.setOrderStatus(OrderStatus.WAIT_PAY.getCode());
        order.setCreateTime(System.currentTimeMillis());
        return order;
    }

    private String generateOrderId(String fsId, String uid) {
        // F + 时间戳(12) + 用户ID后4位 + 随机(6)
        String ts = String.valueOf(System.currentTimeMillis());
        String uidSuffix = uid.length() > 4 ? uid.substring(uid.length() - 4) : uid;
        String rand = UUID.randomUUID().toString().replace("-", "").substring(0, 6);
        return "F" + ts + uidSuffix + rand;
    }
}
```

```java
// FlashOrderMapper.java — 增加批量写方法
@Mapper
public interface FlashOrderMapper extends BaseMapper<FlashOrder> {

    /**
     * INSERT ... ON DUPLICATE KEY UPDATE
     * 利用 (uid, fs_id) 唯一键防重
     *
     * 如果已存在：更新 request_id 和 order_status（保留已有）
     * 如果不存在：插入新记录
     */
    @Insert(
        "INSERT INTO flash_order (request_id, order_id, fs_id, uid, goods_info_id, " +
        "flash_price, qty, order_status, create_time) " +
        "VALUES (#{requestId}, #{orderId}, #{fsId}, #{uid}, #{goodsInfoId}, " +
        "#{flashPrice}, #{qty}, #{orderStatus}, #{createTime}) " +
        "ON DUPLICATE KEY UPDATE " +
        "request_id = VALUES(request_id), " +
        "order_status = IF(order_status = 1, order_status, VALUES(order_status))"
    )
    @Options(useGeneratedKeys = true, keyProperty = "id")
    int insertWithUniqueCheck(FlashOrder order);

    /**
     * 批量 INSERT IGNORE（适用于批量场景）
     * 忽略重复行，不影响其他行
     */
    @Insert(
        "<script>" +
        "INSERT IGNORE INTO flash_order (request_id, order_id, fs_id, uid, goods_info_id, " +
        "flash_price, qty, order_status, create_time) VALUES " +
        "<foreach collection='list' item='o' separator=','>" +
        "(#{o.requestId}, #{o.orderId}, #{o.fsId}, #{o.uid}, #{o.goodsInfoId}, " +
        "#{o.flashPrice}, #{o.qty}, #{o.orderStatus}, #{o.createTime})" +
        "</foreach>" +
        "</script>"
    )
    int batchInsertIgnore(@Param("list") List<FlashOrder> orders);

    // 其他方法 ...
}
```

## 六、唯一键防重机制详解

### 6.1 DB 层三重保护

```
第1层: uk_request_id 唯一键
  └── 同一 requestId 只能 insert 一次
  └── 消息重复 → DuplicateKeyException → 静默忽略

第2层: uk_uid_fs 唯一键
  └── 同一用户对同一活动只能有一条订单
  └── 网络重试、用户重复点击 → 唯一键保证

第3层: ON DUPLICATE KEY UPDATE
  └── 冲突时更新 order_status（保留已支付状态）
  └── 防止 "待支付" 覆盖 "已支付"
```

### 6.2 场景覆盖

| 场景 | 唯一键 | 结果 |
|---|---|---|
| **MQ 消息重投**（同 requestId） | `uk_request_id` | INSERT 失败 → 已存在 → 忽略 |
| **用户重复请求**（同 uid + fsId） | `uk_uid_fs` | INSERT 失败 → 已存在 → 忽略 |
| **网络重试, 前一请求已成功** | `uk_uid_fs` | 重复订单被阻止 |
| **取消后重新下单** | `uk_uid_fs` | 需求确定是否允许；允许则在 `ON DUPLICATE KEY UPDATE` 中重置状态 |

### 6.3 与 Redis 预扣库存的关系

```
          Redis 预扣库存（Lua）
               │
               │ 成功
               ▼
          MQ 消息投递
               │
               │ 可能丢消息、重复、乱序
               ▼
          DB 订单建立（唯一键防重）
               │
               ├── 新订单 → 插入成功 → 扣真实库存
               └── 重复订单 → 唯一键冲突 → 跳过（库存已在 Redis 层控制）

         库存一致性保证：
           Redis 预扣 ≈ 流量控制，防止超卖
           DB 真实库存 ≈ 最终数据底盘
           两者最终一致，以 DB 为准
```

## 七、事务与异常处理

### 7.1 事务边界

```java
@Transactional(rollbackFor = Exception.class)
public FlashOrderWriteResult batchCreateOrders(List<FlashOrderMessage> messages) {
    // ╔══════════════════════════════════════╗
    // ║        一个事务                      ║
    // ║                                      ║
    // ║  INSERT flash_order (批量)            ║
    // ║  +  唯一键冲突忽略                     ║
    // ║                                      ║
    // ║  UPDATE flash_sale (批量)             ║
    // ║  +  乐观锁防超卖                       ║
    // ╚══════════════════════════════════════╝
    //    │
    //    ├── 全部成功 → COMMIT
    //    ├── 部分失败 → ROLLBACK（整批）
    //    └── 唯一键冲突 → 不抛异常，计入 duplicateCount
}
```

### 7.2 异常分类处理

| 异常类型 | 处理方式 | 说明 |
|---|---|---|
| `DuplicateKeyException` | 忽略，计入 duplicateCount | 唯一键保护，正常业务 |
| `DataIntegrityViolationException` | 忽略，计入 duplicateCount | 数据约束冲突 |
| `OptimisticLockException` | 整批回滚，降级逐条重试 | 库存乐观锁冲突 |
| `DataAccessException` | 重试 3 次，仍失败入死信 | DB 连接异常 |
| `Exception` | 入死信队列，人工介入 | 未知异常 |

## 八、性能评估

### 8.1 批量写入 vs 单条写入

```
TPS
8000 │                                    ● 批量 INSERT IGNORE (100条/批)
     │                                  ✦
6000 │                                ✦
     │                              ✦
4000 │                           ✦
```

```java
@Component
public class BatchWriteMonitor {

    private final Timer   batchTimer;

    private final Counter stockDeductTotal;
    private final Counter stockDeductFail;

    public BatchWriteMonitor(MeterRegistry registry) {
        this.registry = registry;

        consumeTotal    = registry.counter("flash.db.consume.total");
        consumeSuccess  = registry.counter("flash.db.consume.success");
        consumeDuplicate = registry.counter("flash.db.consume.duplicate");
        consumeFail     = registry.counter("flash.db.consume.fail");

        batchTotal      = registry.counter("flash.db.batch.total");
        batchTimer      = registry.timer("flash.db.batch.time");
```

### 9.2 告警阈值

| 指标 | 告警阈值 | 说明 |
|---|---|---|
| 消费失败率 | > 1% | 批量写入异常 |
| 唯一键冲突率 | > 10% | 重复消息过多，可能 producer 重试过度 |
| 库存扣减失败 | > 0 | Redis 预扣和 DB 库存不一致 |
| 批量写入耗时 | > 1000ms | DB 性能下降 |
| 堆内缓冲区积压 | > 5000 | 消费速度 < 生产速度 |

## 十一、优化前后对比

| 维度 | 优化前 | 优化后 |
|---|---|---|
| **SQL 防重** | `SELECT` 查一次 → `INSERT` | `INSERT ON DUPLICATE KEY` 一步完成 |
| **唯一键** | 无 | `uk_uid_fs` + `uk_request_id` 双层 |
| **写入方式** | 逐条 insert | 批量 IGNORE / 100 条 |
| **写入 TPS** | ~1200 | ~8000（6.6x） |
| **事务粒度** | 单条事务 | 批量事务 |
| **库存扣减** | 无 DB 层兜底 | 乐观锁 `sold_stock` 实时扣减 |
| **重复消息** | DB 查询判断，有并发窗口 | 唯一键底层保证，100% 防重 |
| **缓存一致性** | 无 | Redis 预扣 + DB 最终一致 |
