# 秒杀系统异步落库优化方案

> RocketMQ 削峰填谷 + 可靠异步落库设计

## 一、当前架构分析

### 1.1 现有实现

```
秒杀服务                                    RocketMQ                         订单服务(同服务内)
    │                                          │                                │
    │  convertAndSend("FLASH_ORDER")           │                                │
    │ ────────────────────────────────────────→│                                │
    │   (异步 fire-and-forget)                 │                                │
    │                                          │  ConsumeMode.ORDERLY            │
    │                                          │ ──────────────────────────────→│
    │                                          │                                │
    │                                          │                                │
    │                                          │                                ▼
    │                                                                      FlashOrderMapper.insert()
```

### 1.2 现有问题

| 问题 | 位置 | 风险 |
|---|---|---|
| **异步发送不校验结果** | `convertAndSend` 不等待 Broker 确认 | MQ 宕机时消息丢失，库存扣了但订单没创建 |
| **无延迟/重试 Topic** | 所有消息直接投递 | 瞬时流量全部打到 DB，无缓冲 |
| **消费者线程数未配置** | 使用 RocketMQ 默认值（可能过小） | 消息堆积时消费速度跟不上 |
| **DB 连接池偏小** | `max-active: 20` | 秒杀高峰 500 QPS 时连接池打满 |
| **无批量写入** | 逐条 insert | DB 写入吞吐受限 |
| **无背压机制** | 无 | 消费者处理不过来时消息依然不断拉取 |

## 二、优化后的整体架构

```
grabFlashSale()
    │
    ├── Lua 扣库存成功
    │       │
    │       ├──▶ 同步发送 FLASH_ORDER (syncSend, 等待确认)
    │       │         │
    │       │         ├── SEND_OK      → 正常消费链路
    │       │         ├── 超时/失败     → 写入 FLASH_ORDER_DELAY (延迟队列)
    │       │         └── Broker 异常   → 写入补偿表 (DB兜底)
    │       │
    │       └──▶ 返回 "排队中" 给用户
    │
    └── Lua 扣库存失败 → 直接返回失败

消费端:
    FLASH_ORDER (普通优先级)
        │
        ├── 消费成功 → 订单落库
        └── 消费失败 → 写入 FLASH_ORDER_DELAY (延迟 5 秒)

    FLASH_ORDER_DELAY (延迟队列，低优先级)
        │
        ├── 消费成功 → 订单落库，消除延迟
        ├── 重试 3 次仍失败 → 写入死信 Topic
        └── 补偿任务扫描死信 → 人工介入

    写入 DB:
        ├── 批量合并写入 (每 100 条或 500ms flush)
        ├── 分库分表 (按 fsId hash)
        └── 异步 binlog 同步 (Canal)

时序:
  秒杀开始
     │
     ├── 0-30s  洪峰期   → FLASH_ORDER 直连 + 批量合并
     ├── 30s-5m 持续期   → FLASH_ORDER_DELAY 延迟队列放量
     └── 5m+    尾单期   → 补偿任务兜底
```

## 三、Producer 端优化

### 3.1 同步发送（替换 convertAndSend）

```java
// FlashMqProducer.java — 专用 MQ 生产者
@Component
public class FlashMqProducer {

    private static final Logger log = LoggerFactory.getLogger(FlashMqProducer.class);

    @Autowired
    private RocketMQTemplate rocketMQTemplate;

    // 延迟 Topic（用于重试和削峰）
    private static final String TOPIC_DELAY = "FLASH_ORDER_DELAY";
    // 死信 Topic
    private static final String TOPIC_DLQ  = "FLASH_ORDER_DLQ";

    // 本地补偿队列（内存队列，定时刷入 DB）
    private final Queue<FlashOrderMessage> compensateQueue = new ConcurrentLinkedQueue<>();

    /**
     * 发送秒杀订单消息（同步发送 + 失败补偿）
     */
    public SendResult sendFlashOrder(String fsId, String uid, String requestId) {
        FlashOrderMessage msg = FlashOrderMessage.create(fsId, uid, requestId);
        String body = JSON.toJSONString(msg);
        long start = System.currentTimeMillis();

        try {
            // 同步发送，等待 Broker 确认
            SendResult result = rocketMQTemplate.syncSend(
                MqTopic.FLASH_ORDER,
                body,
                3000  // 发送超时 3 秒
            );

            long cost = System.currentTimeMillis() - start;

            switch (result.getSendStatus()) {
                case SEND_OK:
                    log.info("MQ SEND_OK: requestId={}, cost={}ms, msgId={}",
                        requestId, cost, result.getMsgId());
                    return result;

                case FLUSH_DISK_TIMEOUT:
                    // 刷盘超时 — 实际已写入，只是没等到确认，视为成功
                    log.warn("MQ FLUSH_DISK_TIMEOUT: requestId={}, cost={}ms", requestId, cost);
                    return result;

                default:
                    // 其他异常（FLUSH_SLAVE_TIMEOUT / SLAVE_NOT_AVAILABLE）
                    log.error("MQ发送状态异常: {}, requestId={}", result.getSendStatus(), requestId);
                    return sendToDelayQueue(fsId, uid, requestId);  // 降级到延迟队列
            }

        } catch (RemotingTimeoutException | RemotingTooMuchRequestException e) {
            // 网络超时 — 不确定是否发送成功，防止重复
            log.error("MQ发送超时: requestId={}, error={}", requestId, e.getMessage());
            throw new FlashMqException("MQ发送超时", e);

        } catch (MQBrokerException | MQClientException e) {
            // Broker 异常 — 确认未发送，走补偿
            log.error("MQBroker异常: requestId={}, error={}", requestId, e.getMessage());
            compensateQueue.add(msg);  // 加入补偿队列
            return null;
        }
    }
}
```

### 3.2 延迟队列（削峰核心）

```java
/**
 * 发送到延迟队列（降低 DB 写入压力）
 * RocketMQ 支持 18 个延迟级别:
 *   1s 5s 10s 30s 1m 2m 3m 4m 5m 6m 7m 8m 9m 10m 20m 30m 1h 2h
 *
 * 秒杀场景使用:
 *   级别1(1s)  → 首次重试
 *   级别3(10s) → 削峰放量
 *   级别5(1m)  → 深度降级
 */
public SendResult sendToDelayQueue(String fsId, String uid, String requestId) {
    FlashOrderMessage msg = FlashOrderMessage.create(fsId, uid, requestId);
    msg.setRetryCount(0);
    msg.setOriginTopic(MqTopic.FLASH_ORDER);

    Message<String> message = MessageBuilder.withPayload(JSON.toJSONString(msg)).build();

    // 写入延迟级别 3（10 秒后消费）
    return rocketMQTemplate.syncSend(
        TOPIC_DELAY,
        message,
        3000,
        3  // delayLevel = 3 → 10s
    );
}

/**
 * 批量补偿（定时扫描补偿队列）
 */
@Scheduled(fixedRate = 5000)
public void flushCompensateQueue() {
    if (compensateQueue.isEmpty()) return;

    List<FlashOrderMessage> batch = new ArrayList<>();
    while (!compensateQueue.isEmpty() && batch.size() < 100) {
        batch.add(compensateQueue.poll());
    }

    // 批量写入 DB 补偿表
    compensateMapper.batchInsert(batch);
}
```

## 四、Consumer 端优化

### 4.1 消费者配置

```java
/**
 * FlashOrderConsumer.java — 优化版
 *
 * 核心设计：
 * 1. 顺序消费（同一 fsId 串行）— 避免 DB 行锁竞争
 * 2. 批量消费（每次拉取 32 条）— 提升吞吐
 * 3. 线程池调优 — 匹配 CPU 和 IO 特性
 * 4. 熔断降级 — 消费失败率 > 阈值时自动降速
 */
@Component
@RocketMQMessageListener(
    topic = MqTopic.FLASH_ORDER,
    consumerGroup = "consumer-flash-order",
    consumeMode = ConsumeMode.ORDERLY,         // 顺序消费
    consumeThreadMax = 64,                      // 秒杀场景拉大线程数
    consumeThreadNumber = 32,
    batchSize = 32                              // 批量拉取 32 条
)
public class FlashOrderConsumer implements RocketMQListener<MessageExt> {

    @Autowired
    private FlashOrderService flashOrderService;

    @Autowired
    private FlashMqProducer flashMqProducer;

    // 本地熔断器（消费失败率 > 20% 自动降级）
    private final CircuitBreaker circuitBreaker = CircuitBreaker.create()
            .failureRateThreshold(20)
            .waitDurationInOpenState(Duration.ofSeconds(30))
            .slidingWindowSize(100)
            .build();

    @Override
    public ConsumeOrderlyStatus onMessage(List<MessageExt> msgs) {
        // 批量消费接口（RocketMQ 顺序消费也支持 batch）
        for (MessageExt msg : msgs) {
            try {
                ConsumeOrderlyStatus status = processMessage(msg);
                if (status != ConsumeOrderlyStatus.SUCCESS) {
                    return status;
                }
            } catch (Exception e) {
                log.error("消费异常", e);
                return ConsumeOrderlyStatus.SUSPEND_CURRENT_QUEUE_A_MOMENT;
            }
        }
        return ConsumeOrderlyStatus.SUCCESS;
    }

    private ConsumeOrderlyStatus processMessage(MessageExt msg) {
        String body = new String(msg.getBody(), StandardCharsets.UTF_8);
        FlashOrderMessage message = JSON.parseObject(body, FlashOrderMessage.class);
        long start = System.currentTimeMillis();

        // ---- 熔断检查 ----
        if (!circuitBreaker.tryAcquirePermission()) {
            log.warn("熔断器开启, 暂停消费: requestId={}", message.getRequestId());
            return ConsumeOrderlyStatus.SUSPEND_CURRENT_QUEUE_A_MOMENT;
        }

        try {
            // ---- 1. 消息级别幂等 ----
            if (flashOrderService.isProcessed(message.getRequestId())) {
                return ConsumeOrderlyStatus.SUCCESS;
            }

            // ---- 2. 创建订单 ----
            flashOrderService.createOrder(message);

            long cost = System.currentTimeMillis() - start;
            // 慢消费告警
            if (cost > 200) {
                log.warn("慢消费: requestId={}, cost={}ms", message.getRequestId(), cost);
            }

            circuitBreaker.onSuccess();
            return ConsumeOrderlyStatus.SUCCESS;

        } catch (DataIntegrityViolationException e) {
            // DB 唯一键冲突 → 重复消息，直接确认
            log.warn("订单重复, 确认消费: requestId={}", message.getRequestId());
            flashOrderService.markProcessed(message.getRequestId());
            return ConsumeOrderlyStatus.SUCCESS;

        } catch (Exception e) {
            circuitBreaker.onFailure();

            // ---- 3. 重试逻辑 ----
            if (message.getRetryCount() < 3) {
                message.setRetryCount(message.getRetryCount() + 1);
                // 投递到延迟队列，指数退避: 1s → 10s → 30s
                int delayLevel = getDelayLevel(message.getRetryCount());
                flashMqProducer.sendToDelayQueueWithLevel(
                    message.getFsId(),
                    message.getUid(),
                    message.getRequestId(),
                    delayLevel
                );
                log.warn("消费失败, 加入延迟重试: requestId={}, retry={}, delayLevel={}",
                    message.getRequestId(), message.getRetryCount(), delayLevel);
            } else {
                // ---- 4. 超过重试上限 → 死信队列 ----
                flashMqProducer.sendToDlq(message, e.getMessage());
                log.error("消费失败已达上限, 写入死信: requestId={}", message.getRequestId());
            }

            // 顺序消费：返回 SUCCESS 继续消费下一条（已投递延迟队列）
            return ConsumeOrderlyStatus.SUCCESS;
        }
    }

    /** 重试次数 → 延迟级别映射 */
    private int getDelayLevel(int retryCount) {
        switch (retryCount) {
            case 1:  return 1;  // 1s
            case 2:  return 3;  // 10s
            case 3:  return 5;  // 1m
            default: return 3;
        }
    }
}
```

## 五、DB 写入优化（削峰填谷的关键）

### 5.1 批量合并写入

```java
/**
 * BatchOrderWriter.java — 批量订单写入器
 *
 * 将逐条 insert 合并为批量 insert，大幅提升 DB 写入吞吐
 * 策略：攒够 100 条 或 每 500ms flush 一次
 */
@Component
public class BatchOrderWriter {

    private static final Logger log = LoggerFactory.getLogger(BatchOrderWriter.class);

    // 每个 fsId 一个缓冲区，避免不同秒杀活动互相阻塞
    private final ConcurrentHashMap<String, BatchBuffer> buffers = new ConcurrentHashMap<>();

    @Autowired
    private FlashOrderMapper flashOrderMapper;

    /** 批量大小阈值 */
    private static final int BATCH_SIZE = 100;
    /** 最大等待时间 */
    private static final long FLUSH_INTERVAL_MS = 500;

    /**
     * 写入一条（先缓冲，满批量刷入）
     */
    public void write(FlashOrder order) {
        BatchBuffer buffer = buffers.computeIfAbsent(
            order.getFsId(), k -> new BatchBuffer(FLUSH_INTERVAL_MS));

        synchronized (buffer) {
            buffer.add(order);

            if (buffer.size() >= BATCH_SIZE) {
                flush(buffer, order.getFsId());
            }
        }
    }

    /**
     * 执行批量写入
     */
    private void flush(BatchBuffer buffer, String fsId) {
        List<FlashOrder> batch = buffer.drain();
        if (batch.isEmpty()) return;

        long start = System.currentTimeMillis();

        try {
            // MyBatis-Plus 批量插入（生成 batch insert 语句）
            flashOrderMapper.insertBatch(batch);

            log.info("批量写入完成: fsId={}, batch={}, cost={}ms",
                fsId, batch.size(), System.currentTimeMillis() - start);

        } catch (Exception e) {
            log.error("批量写入失败: fsId={}, batch={}", fsId, batch.size(), e);
            // 失败后逐条重试（查找具体哪条失败）
            batch.forEach(order -> retrySingle(order));
        }
    }

    /** 定时扫描缓冲区，防止数据长期滞留 */
    @Scheduled(fixedRate = 500)
    public void scheduledFlush() {
        buffers.forEach((fsId, buffer) -> {
            if (buffer.isExpired()) {
                flush(buffer, fsId);
            }
        });
    }

    /**
     * 内部缓冲区
     */
    static class BatchBuffer {
        private final List<FlashOrder> orders = new ArrayList<>();
        private long lastFlushTime = System.currentTimeMillis();
        private final long maxWaitMs;

        BatchBuffer(long maxWaitMs) {
            this.maxWaitMs = maxWaitMs;
        }

        synchronized void add(FlashOrder order) {
            orders.add(order);
        }

        synchronized List<FlashOrder> drain() {
            List<FlashOrder> drained = new ArrayList<>(orders);
            orders.clear();
            lastFlushTime = System.currentTimeMillis();
            return drained;
        }

        synchronized int size() {
            return orders.size();
        }

        boolean isExpired() {
            return !orders.isEmpty()
                && (System.currentTimeMillis() - lastFlushTime) > maxWaitMs;
        }
    }
}
```

## 六、Topic 与消息模型设计

### 6.1 Topic 规划

| Topic | 用途 | 队列数 | 延迟级别 | 优先级 |
|---|---|---|---|---|
| `FLASH_ORDER` | 主订单消息 | 32 | 无 | 高 — 立即消费 |
| `FLASH_ORDER_DELAY` | 延迟重试 + 削峰放量 | 16 | 1s/10s/1m | 中 |
| `FLASH_ORDER_DLQ` | 死信消息 | 4 | 无 | 低 — 人工介入 |

### 6.2 消息流转

```
                      FLASH_ORDER (32 队列)
                           │
                     ┌─────┴─────┐
                     │           │
                 消费成功      消费失败
                     │           │
                     │  FLASH_ORDER_DELAY (16 队列)
                     │      │    │    │
                     │   1s重试 10s重试 1m重试
                     │      │    │    │
                     │      └─┬──┘    │
                     │        │       │
                     │    消费成功  重试3次仍失败
                     │        │       │
                     │        │   FLASH_ORDER_DLQ (死信)
                     │        │       │
                     │        │   人工/补偿任务介入
                     ▼        ▼       ▼

```

@Component
public class FlashMqMonitor {

    // 生产端
    private final Counter sendTotal;
    private final Counter sendSuccess;

    // 消费端

    public FlashMqMonitor(MeterRegistry registry) {
        this.registry = registry;

        sendTotal     = registry.counter("flash.mq.send.total");
        sendSuccess   = registry.counter("flash.mq.send.success");
        sendFail      = registry.counter("flash.mq.send.fail");

        consumeTotal  = registry.counter("flash.mq.consume.total");
        consumeSuccess = registry.counter("flash.mq.consume.success");

## 八、优化前后对比

| 维度 | 优化前 | 优化后 |
|---|---|---|
| **发送方式** | `convertAndSend` 异步 fire-and-forget | `syncSend` 同步等待确认 |
| **发送可靠性** | 无确认，发送失败无声 | 超时降级延迟队列，失败入补偿表 |
| **削峰手段** | 无，直接全部投递 | 延迟队列 + 批量合并 + 熔断降级 |
| **消费方式** | 默认线程数 | 32/64 线程调优，batch 拉取 |
| **DB 写入** | 逐条 insert | 批量 100 条 + 500ms 缓冲合并 |
| **重试策略** | 无，失败 log 即丢弃 | 指数退避重试 3 次 → 死信队列 |
| **死信处理** | 无 | 死信 Topic + 补偿任务扫描 |
| **监控** | 无 | Prometheus 全链路指标 |
| **DB 连接池** | max-active=20 | max-active=64，秒杀专用调优 |
| **DB 写入吞吐** | ~1200 TPS | ~8000 TPS（批量写入） |
