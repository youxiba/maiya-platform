# 秒杀系统高并发设计方案

## 一、现状分析

当前秒杀实现的问题（基于现有 `maiya-marketing-service`）：

| 问题 | 当前 | 理想 |
|------|------|------|
| 库存扣减 | MySQL 直写 `UPDATE goods_sku SET sum=sum-1` | Redis 原子扣减 + 异步落库 |
| 流量削峰 | 无 | MQ 排队，数据库按单处理 |
| 限流 | 无 | 令牌桶限流，拦截刷单 |
| 页面 | 动态渲染 | 页面静态化到 CDN |
| 重复下单 | 无防护 | 用户级去重（Redis + 唯一索引） |

## 三、核心流程设计

### 阶段 1：秒杀前 - 商品预热

```java
// MarketingService 启动时或定时任务执行
@PostConstruct
public void preheatFlashSale() {
    List<FlashSale> activeSales = flashSaleMapper.findActive();
    for (FlashSale sale : activeSales) {
        String stockKey = "flash:stock:" + sale.getFsId();
        String userSetKey = "flash:users:" + sale.getFsId();
        
        // 从 DB 加载库存到 Redis
        int totalStock = goodsSkuMapper.sumStockByActive(sale.getFsId());
        redisTemplate.opsForValue().set(stockKey, String.valueOf(totalStock));
        
        // 设置活动标记
        redisTemplate.opsForValue().set("flash:active:" + sale.getFsId(), "1",
            30, TimeUnit.MINUTES); // 活动存活时间
    }
}
```

### 阶段 2：秒杀中 - 流量控制

```
用户请求秒杀
    │
    ▼
① Gateway 限流 (Sentinel)
    ├── 全局: 1000 QPS
    └── 用户级别: 10 QPS
    │
    ▼
② 前置校验 (Marketing Service)
    ├── 活动是否进行中 (Redis)
    ├── 用户是否已购买 (Redis Set)
    ├── 用户是否在黑名单 (Redis)
    └── 限购数量校验 (Redis incr)
    │
    ▼
③ Lua 脚本扣库存 (Redis 原子操作)
    │
    ▼
④ 发送 MQ 订单请求 (快速返回)
    └── 返回 "排队中" 给用户
```

### 阶段 3：Redis Lua 库存扣减（核心）

```lua
-- flash_stock.lua
-- KEYS[1] = flash:stock:{fsId}     库存 key
-- KEYS[2] = flash:users:{fsId}     已购买用户 set
-- ARGV[1] = uid                    用户ID
-- ARGV[2] = limit                  限购数量

-- 1. 检查用户是否已购买
local purchased = redis.call('SISMEMBER', KEYS[2], ARGV[1])
if purchased == 1 then
    return -2  -- 重复购买
end

-- 2. 扣减库存
local stock = redis.call('DECR', KEYS[1])
if stock < 0 then
    redis.call('INCR', KEYS[1])  -- 回滚
    return -1  -- 库存不足
end

-- 3. 记录已购买用户
redis.call('SADD', KEYS[2], ARGV[1])

-- 4. 设置 key 过期时间（匹配活动时长）
redis.call('EXPIRE', KEYS[2], 86400)

return stock  -- 返回剩余库存
```

## 四、数据表设计（新增）

```sql
-- 秒杀订单表（独立于普通订单，高频写入）
CREATE TABLE flash_order (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    order_id VARCHAR(64) NOT NULL,
    fs_id VARCHAR(64) NOT NULL COMMENT '秒杀活动ID',
    uid VARCHAR(64) NOT NULL,
    goods_info_id VARCHAR(64) NOT NULL,
    flash_price DECIMAL(19,2) NOT NULL,
    qty INT DEFAULT 1,
    order_status TINYINT DEFAULT 0 COMMENT '0=待支付 1=已支付 2=已取消',
    create_time BIGINT NOT NULL,
    pay_time BIGINT DEFAULT 0,
    UNIQUE KEY uk_uid_active (uid, fs_id)  -- 一人一单
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 五、防超卖方案对比

| 方案 | 性能 | 安全性 | 复杂度 | 适用 |
|------|------|--------|--------|------|
| ✅ **Redis Lua** | 极高 (1ms) | ✅ 原子操作 | 低 | 本方案 |
| ❌ MySQL 乐观锁 | 中 (10ms) | ✅ 行锁 | 低 | 当前方案 |
| ❌ MySQL 悲观锁 | 低 (50ms) | ✅ | 低 | 不推荐 |
| ❌ 分布式锁+MySQL | 低 (20ms) | ⚠️ 锁超时 | 高 | 不推荐 |

## 六、防刷设计

```
┌─────────────────────────────────────────────────────────────────┐
│  防刷手段                                                        │
│                                                                  │
│  1. 用户级别限流 (Sentinel) - 10 QPS/用户                       │

│  4. Redis 黑名单 - 检测到刷单行为加入黑名单                     │
│  5. 异步排队 - 不直接返回成功，统一进 MQ 排队                    │
│  6. 一人一单 - (uid, fsId) 唯一索引 + Redis Set                 │
└─────────────────────────────────────────────────────────────────┘
```

## 七、库存回滚

用户支付超时或取消订单时，需要回滚库存：

```java
// 订单超时 → 回滚 Redis 库存 + 移除已购买标记
public void rollbackStock(String fsId, String uid, int qty) {
    String stockKey = "flash:stock:" + fsId;
    String userSetKey = "flash:users:" + fsId;
    
    // Lua 回滚脚本
    redisTemplate.opsForValue().increment(stockKey, qty);
    redisTemplate.opsForSet().remove(userSetKey, uid);
}
```
