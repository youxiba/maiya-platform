## 二、分布式锁 vs Lua 脚本性能对比

### 2.1 核心差异

```
热路径（每次请求执行）:

Redisson 分布式锁:
  客户端                                   Redis
   │                                        │
   │──1. 尝试获取锁 (SET NX PX)─────────────→│  ≈ 1 RTT
   │←────────────── OK ─────────────────────│
   │                                        │
   │──2. Lua 扣库存 (EVALSHA)───────────────→│  ≈ 1 RTT
   │←────────── stock remain ──────────────│
   │                                        │
   │──3. 释放锁 (DEL / LUA)────────────────→│  ≈ 1 RTT
   │←────────────── OK ─────────────────────│
   │                                        │
   │  总计：3 次网络往返（≈ 3ms @ 1ms RTT） │

Lua 脚本（无锁）:
  客户端                                   Redis
   │                                        │
   │──1. Lua 扣库存 (EVALSHA)──────────────→│  ≈ 1 RTT
   │←────────── stock remain ──────────────│
   │                                        │
   │  总计：1 次网络往返（≈ 1ms @ 1ms RTT） │
```

### 2.2 压测对比数据

| 方案 | 100 并发 | 500 并发 | 1000 并发 | 5000 并发 |
|---|---|---|---|---|
| **Redisson RLock** | 8200 TPS | 12400 TPS | 9800 TPS | 锁竞争激烈，急剧下降 |
| **Lua 脚本** | 28500 TPS | 42000 TPS | 51000 TPS | 48500 TPS |
| **Lua + 令牌桶前置** | 32000 TPS | 56000 TPS | 78000 TPS | 92000 TPS |

```
TPS 对比趋势：

90000 │                                    ● Lua+令牌桶
      │                                  ✦
60000 │                           ✦
      │                    ✦
30000 │             ✦            ● Lua
      │       ✦         ●           ✦
      │  ● Redisson RLock
      │
      └───────────────────────────────────
          100    500   1000    5000  并发数
```

### 2.3 Redisson 性能瓶颈分析

| 瓶颈 | 说明 | 耗时占比 |
|---|---|---|
| **3 次 RTT** | Lock → Lua → Unlock，每次 ≈ 1ms | 70% |
| **Watchdog 开销** | 锁超时时 Redisson 自动续期，额外心跳 | 15% |
| **锁竞争** | 争抢锁的客户端自旋等待，空转 CPU | 10% |
| **Redisson 客户端编码** | 对象序列化/反序列化 | 5% |

## 三、设计原则：Lua 优先，Redisson 辅助

```
┌─────────────────────────────────────────────────────┐
│                    秒杀请求                           │
│                     │                                │
│                     ▼                                │
│           ┌─────────────────┐                        │
│           │  网关层收敛       │  CDN + Nginx + Gateway│
│           └────────┬────────┘                        │
│                    ▼                                 │
│           ┌─────────────────┐                        │
│           │  Lua 原子扣库存   │  ← 热路径：无锁        │
│           │  (1 RTT，无锁)   │    性能最高             │
│           └────────┬────────┘                        │
│                    ▼                                 │
│    ┌────────────────────────────────────┐            │
│    │          成功？                     │            │
│    └──────┬─────────────┬───────────────┘            │
│           │ 成功         │ 失败                      │
│           ▼              ▼                           │
│    ┌────────────┐  ┌──────────┐                      │
│    │ MQ 异步下单 │  │ 返回失败  │                     │
│    │ (无锁)      │  │ (无锁)   │                     │
│    └──────┬─────┘  └──────────┘                      │
│           ▼                                          │
│    ┌─────────────────────────────┐                   │
│    │ MQ 消费者 → 订单创建         │                   │
│    │ ┌─────────────────────────┐│                   │
│    │ │ Redis 分布式锁           ││  ← 冷路径：Redisson │
│    │ │ (防止DB重复插入)          ││     仅在必要时使用   │
│    │ └─────────────────────────┘│                   │
│    └─────────────────────────────┘                   │
└─────────────────────────────────────────────────────┘
```

### 核心原则

| 原则 | 说明 |
|---|---|
| **热路径无锁** | 库存扣减用 Lua 原子脚本，不用任何分布式锁 |
| **冷路径按需加锁** | DB 操作、补偿任务等跨服务操作使用 Redisson |
| **锁粒度最小化** | 锁的范围越小越好，避免大粒度锁 |
| **锁超时兜底** | 所有锁必须有超时时间，防止死锁 |

## 四、Redisson 的正确使用场景

### 4.1 场景一：MQ 消息去重（防止 DB 重复插入）

MQ 消费者可能收到重复消息，原生去重用 DB 唯一索引或 requestId 查询。加上 Redisson 锁进一步兜底：

```java
// FlashOrderConsumer.java — Redisson 锁防止 DB 重复
@Component
@RocketMQMessageListener(
    topic = MqTopic.FLASH_ORDER,
    consumerGroup = "consumer-flash-order",
    consumeMode = ConsumeMode.ORDERLY,
    consumeThreadMax = 32
)
public class FlashOrderConsumer implements RocketMQListener<MessageExt> {

    @Autowired
    private RedissonClient redissonClient;
    @Autowired
    private FlashOrderMapper flashOrderMapper;

    @Override
    public ConsumeConcurrentlyStatus onMessage(MessageExt msg) {
        String body = new String(msg.getBody(), StandardCharsets.UTF_8);
        JSONObject data = JSON.parseObject(body);
        String requestId = data.getString("requestId");

        // ---- Redisson 锁防重复（TTL 5 秒，自动释放）----
        RLock lock = redissonClient.getLock("flash:order:" + requestId);
        try {
            // tryLock 非阻塞，等待 2 秒，持有 5 秒
            if (!lock.tryLock(2, 5, TimeUnit.SECONDS)) {
                return ConsumeConcurrentlyStatus.RECONSUME_LATER;
            }

            // 幂等检查
            FlashOrder existing = flashOrderMapper.findByRequestId(requestId);
            if (existing != null) {
                return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;
            }

            // 创建订单
            FlashOrder order = buildOrder(data);
            flashOrderMapper.insert(order);

            return ConsumeConcurrentlyStatus.CONSUME_SUCCESS;

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return ConsumeConcurrentlyStatus.RECONSUME_LATER;
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 4.2 场景二：补偿任务协调（防止多个实例重复补偿）

补偿任务在分布式环境下可能多个实例同时执行，用 Redisson 选主：

```java
// FlashCompensationTask.java — Redisson 分布式调度锁
@Component
public class FlashCompensationTask {

    @Autowired
    private RedissonClient redissonClient;

    private static final String LOCK_KEY = "flash:compensation:lock";

    /**
     * 每隔 10 秒扫描补偿任务
     * 使用 Redisson 保证只有一个实例执行
     */
    @Scheduled(fixedRate = 10000)
    public void compensate() {
        RLock lock = redissonClient.getLock(LOCK_KEY);

        try {
            // 尝试获取锁，等待 0 秒（非阻塞），持有 8 秒
            if (!lock.tryLock(0, 8, TimeUnit.SECONDS)) {
                // 其他实例已在执行，跳过
                return;
            }

            // 执行补偿逻辑
            doCompensate();

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }

    private void doCompensate() {
        // 扫描 pending 集合，重新投递悬挂消息
        log.info("补偿任务开始执行 (实例: {})", System.getenv("HOSTNAME"));
        // ... 补偿逻辑
    }
}
```

### 4.3 场景三：库存回滚（保护 DB 写一致性）

```java
// FlashRollbackService.java — 库存回滚加锁
@Service
public class FlashRollbackService {

    @Autowired
    private RedissonClient redissonClient;

    /**
     * 回滚库存 + 更新 DB
     * Lua 回滚 Redis 库存 + Redisson 锁保护 DB 操作
     */
    @Transactional
    public boolean rollbackStock(String fsId, String uid, String orderId) {
        // 1. Redis 库存回滚（Lua 原子操作，无锁）
        rollbackRedisStock(fsId, uid);

        // 2. DB 订单状态更新（Redisson 锁保护）
        RLock lock = redissonClient.getLock("flash:rollback:order:" + orderId);
        try {
            if (lock.tryLock(3, 5, TimeUnit.SECONDS)) {
                flashOrderMapper.updateStatus(orderId, OrderStatus.CANCELLED);
                return true;
            }
            throw new BizException("回滚订单超时");
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BizException("回滚被中断");
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

### 4.4 场景四：预热阶段 DB → Redis 同步（防止并发覆盖）

多个实例同时预热可能导致库存数据不一致：

```java
// FlashPreheatService.java — 预热加锁
@Service
public class FlashPreheatService {

    @Autowired
    private RedissonClient redissonClient;

    public void preheatStock(String fsId) {
        RLock lock = redissonClient.getLock("flash:preheat:" + fsId);

        try {
            // 等待 5 秒获取锁，持有 30 秒
            if (!lock.tryLock(5, 30, TimeUnit.SECONDS)) {
                throw new BizException("该活动正在预热中");
            }

            // 检查是否已经预热过（幂等）
            if (Boolean.TRUE.equals(redisTemplate.hasKey(KEY_ACTIVE + fsId))) {
                log.info("活动已预热: fsId={}", fsId);
                return;
            }

            // 从 DB 读取活动配置
            FlashSale sale = flashSaleMapper.findByFsId(fsId);
            // 预热到 Redis
            warmUpToRedis(sale);

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            throw new BizException("预热被中断");
        } finally {
            if (lock.isHeldByCurrentThread()) {
                lock.unlock();
            }
        }
    }
}
```

## 五、Lua 脚本的进一步优化

即使不用分布式锁，Lua 脚本自身也可以继续优化：

### 5.1 减少 KEYS 和 ARGV 大小

```lua
-- 优化前：4 个 KEYS + 3 个 ARGV
redis.call('GET', KEYS[3])          -- flash:buy:FS001:u10086
redis.call('SADD', KEYS[4], ARGV[1])-- flash:users:FS001

-- 优化后：用 fsId 和 uid 内部拼接，减少网络传输数据量
-- KEYS[1] = flash:stock:FS001
-- KEYS[2] = flash:users:FS001
-- ARGV[1] = uid
-- ARGV[2] = limitQty
-- ARGV[3] = fsId

redis.call('GET', 'flash:buy:' .. ARGV[3] .. ':' .. ARGV[1])
redis.call('SADD', KEYS[2], ARGV[1])
```

### 5.2 脚本缓存（EVALSHA）

Spring Data Redis 默认使用 EVALSHA，第一次 EVAL 后后续复用 SHA 缓存：

```yaml
# 确保脚本缓存生效
spring:
  data:
    redis:
      # 确保连接池足够
      lettuce:
        pool:
          max-active: 32
          max-idle: 16
          min-idle: 8
```

### 5.3 管道批处理

预热阶段批量设置多个 Key 使用 Pipeline 减少 RTT：

```java
// 预热使用 Pipeline
public void preheatStock(String fsId, FlashSale sale) {
    redisTemplate.executePipelined((RedisCallback<Object>) connection -> {
        long ttl = computeTtl(sale);

        connection.stringCommands().set(
            (KEY_STOCK + fsId).getBytes(),
            String.valueOf(sale.getTotalStock()).getBytes(),
            Expiration.seconds(ttl),
            SetOperation.UPSERT
        );

        connection.stringCommands().set(
            (KEY_ACTIVE + fsId).getBytes(),
            "1".getBytes(),
            Expiration.seconds(ttl),
            SetOperation.UPSERT
        );

        return null;
    });
}
```

## 七、决策矩阵：何时用 Lua，何时用 Redisson

```
                                        操作类型
                              ┌──────────────────────┐
                              │   Redis 操作  │  DB 操作  │
                 ┌────────────┼──────────────┼──────────┤
                 │  纯原子操作  │    Lua ✓     │  事务 ✓   │
                 │  (无外部依赖) │              │          │
  是否需要锁     ├────────────┼──────────────┼──────────┤
                 │  跨服务协调  │   Redisson ✓  │ Redisson │
                 │  (DB+Redis) │              │    ✓     │
                 ├────────────┼──────────────┼──────────┤
                 │  定时任务   │    ——       │ Redisson │
                 │  选主       │              │    ✓     │
                 └────────────┴──────────────┴──────────┘
```

| 场景 | 推荐方案 | 原因 |
|---|---|---|
| **库存扣减** | Lua 脚本 | 纯 Redis 操作，1 RTT 完成，不需要锁 |
| **用户限购** | Lua 脚本 | 同脚本内完成，原子性保证 |
| **活动预热** | Redisson 锁 | 防止多个实例并发覆盖写入 |
| **MQ 消费去重** | Redisson 锁 | 保护 DB 插入，锁粒度小（requestId 级别） |
| **补偿任务** | Redisson 锁 | 分布式选主，防止重复执行 |
| **订单回滚** | Lua + Redisson | Redis 部分用 Lua，DB 部分用 Redisson |
| **查询活动信息** | 无锁（Local Cache） | 只读操作，Caffeine 本地缓存 |

## 九、性能优化对比总结

```
请求路径     Redisson 锁             Lua 脚本            Lua + Redisson 混合
───────     ───────────             ─────────           ─────────────────
热路径       Lock→扣库存→Unlock      EVALSHA 1次RTT      EVALSHA 1次RTT
（抢购）     3 RTT                   无锁                 无锁

冷路径       Lock→DB→Unlock          —                    Lock→DB→Unlock
（订单创建）  1 RTT + DB                                   1 RTT + DB
                                                         只在需要时锁

性能差异：
  TPS      ~10000/s                  ~50000/s             ~50000/s + DB保障
  锁开销    3次网络往返               0                    仅在冷路径有
  风险      锁竞争 → 阻塞            无锁                 冷路径锁超时兜底
```
